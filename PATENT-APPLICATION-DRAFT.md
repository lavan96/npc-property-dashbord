## 2. ABSTRACT

A computer-implemented system and method for automated property investment analysis comprising:

A multi-tier report generation engine that produces comprehensive investment analysis documents through a twelve-section chunked processing architecture, wherein each section is independently generated by artificial intelligence and persisted progressively to enable fault-tolerant resumption without content regeneration. The system maintains narrative consistency across sections by injecting a predetermined character count from previously generated content into subsequent prompts.

A unified data aggregation pipeline that integrates real-time information from at least ten (10) disparate authoritative data sources including government census databases, crime statistics bureaus, meteorological services, property market indices, environmental risk registries, and financial rate providers. Retrieved data is cached with configurable source-specific expiration periods and quality indicators enabling graceful degradation when sources are unavailable.

A client relationship management subsystem that captures multi-dimensional investor profiles including detailed income breakdowns with lender-specific shading factors, categorized expenses with essential/discretionary classification, multi-property portfolios with individual cash flow tracking, and computed borrowing capacity assessments using lender-standard formulas with regulatory stress-testing buffers.

A secure service-mediated database architecture wherein all frontend data operations are routed through authenticated serverless functions that verify user sessions against a session database, retrieve user permissions, and execute database queries using elevated service-role credentials. This architecture prevents direct client-side database access while maintaining row-level security as a secondary enforcement layer.

A conversational document analysis interface utilizing Retrieval-Augmented Generation (RAG) for natural language querying of generated investment reports, with streaming response delivery and persistent conversation history.

An automation engine with configurable trigger switches that automatically generate investment reports when new property listings matching specified criteria are detected, with duplicate prevention and processing history tracking.

The system addresses the technical problems of: (1) reducing property investment due diligence from hours of manual analysis to minutes of automated processing; (2) eliminating data fragmentation across multiple specialized platforms; (3) providing unified client intelligence combining property analysis with borrowing capacity assessment; (4) enabling secure multi-user access to sensitive financial data through a novel mediation architecture; and (5) overcoming serverless execution time limits through chunked progressive processing.

## 4. BACKGROUND OF THE INVENTION

### 4.1 Industry Context

The property investment industry in Australia represents a significant economic sector, with residential property valued at over $10 trillion AUD. Professional buyer's agents, property investment advisors, and mortgage brokers require comprehensive due diligence analysis for each property considered by their clients.

The due diligence process traditionally involves:

1. **Property Data Gathering**: Accessing property portals to retrieve listing information, photos, floor plans, and agent details.

2. **Market Research**: Analyzing comparable sales, suburb median prices, and historical growth trends from specialized data providers.

3. **Location Analysis**: Assessing proximity to amenities, transport, schools, employment centers, and understanding suburb demographics.

4. **Risk Assessment**: Evaluating environmental risks (flooding, bushfire), crime statistics, and infrastructure developments.

5. **Financial Modeling**: Calculating yield, cash flow projections, stamp duty, and ongoing costs.

6. **Client Matching**: Assessing whether the property suits the client's financial circumstances, investment goals, and borrowing capacity.

7. **Report Production**: Compiling findings into a professional report for client presentation.

This process typically requires 4-8 hours per property when performed manually, creating significant scalability constraints for investment advisory practices.

#### Problem 4: Serverless Timeout Constraints

**Current State**:

Complex AI document generation using large language models faces execution time limitations:

| Platform | Typical Timeout | Complex Report Time | Result |
|----------|----------------|---------------------|--------|
| AWS Lambda | 30 seconds (default) | 2-5 minutes | Failure |
| Vercel Functions | 60 seconds | 2-5 minutes | Failure |
| Supabase Edge Functions | 150 seconds | 2-5 minutes | Partial/Failure |
| Cloudflare Workers | 50ms CPU limit | N/A | Unsuitable |

Comprehensive investment reports requiring multiple AI calls for different sections consistently exceed these limits, resulting in:
- Incomplete reports (truncated content)
- Failed generation with no output
- Wasted API costs for abandoned generation
- Manual restart required to complete

**Inventive Solution**:

Chunked processing architecture with progressive persistence and fault-tolerant resumption:

- Report divided into 12 predefined sections
- Each section generated by independent AI call
- Section content persisted immediately upon generation
- Progress tracker records last completed section
- If interrupted, resume from last completed section
- No content regeneration for completed sections

**Technical Implementation**:
- `last_completed_section` column tracking progress (0-11)
- Progressive database updates after each section
- Status tracking (generating/completed/failed)
- Resume logic checking existing progress on restart
- Content clearing logic for fresh starts

#### Problem 5: Secure Multi-User Data Access

**Current State**:

Web applications face security challenges with direct database access:

**Approach 1: Direct Client-Side Database Access**
- Database credentials exposed in frontend code
- SQL injection vulnerabilities
- No centralized authorization logic
- Difficult to audit access patterns

**Approach 2: Complex Row-Level Security (RLS)**
- Policy logic distributed across database
- Difficult to test and maintain
- Limited flexibility for complex authorization rules
- Performance impact from policy evaluation

**Approach 3: Traditional API Servers**
- Additional infrastructure to maintain
- Scaling complexity
- Deployment overhead
- Cost for always-on resources

**Inventive Solution**:

Service-mediated architecture where all data operations flow through authenticated serverless functions:

- Frontend makes no direct database connections
- All operations routed through `invokeSecureFunction()` pattern
- Edge functions verify session and permissions
- Database queries executed with service-role credentials
- RLS maintained as backup defense layer
- Centralized audit logging

**Technical Implementation**:
- 86 edge functions covering all data operations
- Session table with token-based authentication
- Permission table with module-level granularity
- HTTP-only cookies for XSS protection
- Service-role credential isolation

#### Problem 6: Narrative Consistency in Chunked Generation

**Current State**:

When generating documents in multiple AI calls:
- Each call is independent with no memory of previous calls
- Writing style may vary between sections
- Facts mentioned in one section may be contradicted in another
- Document lacks cohesive narrative flow
- Reader experiences jarring transitions

**Inventive Solution**:

Context injection maintaining narrative consistency:

- Extract last 4,000 characters from previously generated content
- Include as context in prompt for next section
- Instruct AI to maintain consistency with provided context
- Reference previous section headings and key points

**Technical Implementation**:
- `getLastNCharacters()` function for context extraction
- Context injection in `buildSectionPrompt()`
- System instructions emphasizing narrative continuity
- Style guide inclusion in prompts

    saveReport(report)

function fetchDataWithCache(source, locationKey):
  // 1. Check cache
  cached = queryCache(source, locationKey)
  
  if cached AND cached.expires_at > now():

#### 5.3.3 Lender-Standard Borrowing Capacity Computation

**Innovation Description**:

The system calculates borrowing capacity using formulas derived from Australian lending practices, incorporating APRA regulatory buffers and lender-specific income shading rules.

**Calculation Methodology**:

**Step 1: Income Aggregation and Shading**

Different income types are "shaded" (reduced) according to lender policies reflecting the reliability and sustainability of each income type:

| Income Type | Shading Factor | Rationale |
|-------------|---------------|-----------|
| Base Salary (PAYG) | 100% | Most reliable income source |
| Base Salary (Casual) | 80% | Less job security |
| Overtime (Essential) | 80% | Reasonably reliable if documented |
| Overtime (Non-Essential) | 50% | May not continue |
| Bonus | 50-80% | Variable, tenure-dependent |
| Commission | 60% | Variable by nature |
| Rental Income (Existing) | 80% | Less expenses/vacancy |
| Rental Income (Proposed) | 80% | Less expenses/vacancy |
| Government Benefits | 100% | Reliable if documented |
| Self-Employment | 60-100% | Depends on business history |

**Step 2: Tax Calculation**

After-tax income calculated using current Australian tax brackets:

```
function calculateAfterTax(grossAnnual):
  // 2025-26 Australian Tax Brackets
  if grossAnnual <= 18200:
    tax = 0
  else if grossAnnual <= 45000:
    tax = (grossAnnual - 18200) * 0.16
  else if grossAnnual <= 135000:
    tax = 4288 + (grossAnnual - 45000) * 0.30
  else if grossAnnual <= 190000:
    tax = 31288 + (grossAnnual - 135000) * 0.37
  else:
    tax = 51638 + (grossAnnual - 190000) * 0.45
  
  // Medicare Levy (2%)
  medicare = grossAnnual * 0.02
  
  return grossAnnual - tax - medicare
```

**Step 3: Expense Assessment**

The system compares declared expenses against HEM (Household Expenditure Measure) benchmarks:

```
function assessExpenses(declaredMonthly, householdSize, location):
  hemBenchmark = getHEMBenchmark(householdSize, location)
  
  // Lenders use higher of declared or HEM
  if declaredMonthly > hemBenchmark:
    return { amount: declaredMonthly, method: 'declared' }
  else:
    return { amount: hemBenchmark, method: 'HEM' }
```

**Step 4: Existing Commitment Assessment**

Existing loan commitments are assessed at stressed rates:

| Commitment Type | Assessment Method |
|-----------------|-------------------|
| Existing Mortgages | P&I at 9.5% assessment rate |
| Personal Loans | Actual repayment |
| Car Loans | Actual repayment |
| Credit Cards | 3% of limit (regardless of balance) |
| HECS/HELP | Repayment based on income threshold |
| Negative Property Cashflow | Full shortfall amount |

**Step 5: Surplus Calculation**

```
function calculateMonthlySurplus(shadedAnnualIncome, monthlyExpenses, monthlyCommitments, propertyNegativeCashflows):
  monthlyNetIncome = calculateAfterTax(shadedAnnualIncome) / 12
  totalExpenses = monthlyExpenses + monthlyCommitments + sum(propertyNegativeCashflows)
  
  return monthlyNetIncome - totalExpenses
```

**Step 6: Maximum Borrowing Calculation**

Using assessment rate (current rate + 3% APRA buffer):

```
function calculateMaxBorrowing(monthlySurplus, assessmentRate, loanTermYears):
  // Present Value of Annuity formula
  monthlyRate = assessmentRate / 12
  numPayments = loanTermYears * 12
  
  maxLoan = monthlySurplus * ((1 - (1 + monthlyRate)^(-numPayments)) / monthlyRate)
  
  return max(0, maxLoan)
```

**Step 7: DTI Ratio and Serviceability Band**

```
function calculateDTI(totalDebt, grossAnnualIncome):
  return totalDebt / grossAnnualIncome

function classifyServiceability(dtiRatio):
  if dtiRatio < 4.0:
    return 'excellent'
  else if dtiRatio < 5.5:
    return 'good'
  else if dtiRatio < 7.0:
    return 'marginal'
  else:
    return 'restricted'
```

// src/lib/supabase-utils.ts

export async function invokeSecureFunction<T>(
  functionName: string,
  payload: Record<string, any> = {}
): Promise<{ data: T | null; error: Error | null }> {
  const sessionToken = sessionStorage.getItem('session_token');
  
  try {
    const { data, error } = await supabase.functions.invoke(functionName, {
      body: { ...payload },
      headers: {
        'x-session-token': sessionToken || '',
        'Content-Type': 'application/json'
      }
    });
    
    if (error) {
      console.error(`Error invoking ${functionName}:`, error);
      return { data: null, error };
    }
    
    return { data: data as T, error: null };
  } catch (err) {
    console.error(`Exception invoking ${functionName}:`, err);
    return { data: null, error: err as Error };
  }
}
```

export async function verifyAuth(req: Request): Promise<AuthResult> {
  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  );
  
  // Extract token from multiple sources
  const cookies = parseCookies(req.headers.get('cookie') || '');
  const headerToken = req.headers.get('x-session-token');
  let body: any = {};
  
  try {
    const text = await req.text();
    if (text) body = JSON.parse(text);
  } catch {}
  
  const sessionToken = headerToken || cookies['session_id'] || body.session_token;
  
  if (!sessionToken) {
    return { success: false, error: 'No session token provided' };
  }
  
  // Validate session
  const { data: session, error: sessionError } = await supabase
    .from('user_sessions')
    .select('user_id, expires_at')
    .eq('session_token', sessionToken)
    .maybeSingle();
  
  if (sessionError || !session) {
    return { success: false, error: 'Invalid session' };
  }
  
  if (new Date(session.expires_at) < new Date()) {
    return { success: false, error: 'Session expired' };
  }
  
  // Get user with permissions
  const { data: user, error: userError } = await supabase
    .from('custom_users')
    .select('*, user_permissions(*)')
    .eq('id', session.user_id)
    .single();
  
  if (userError || !user) {
    return { success: false, error: 'User not found' };
  }
  
  return {
    success: true,
    user: user,
    supabase: supabase  // Service-role client for queries
  };
}
```

function processNewListings():
  // 1. Check master switch
  masterSettings = getMasterSettings()
  if not masterSettings.is_enabled:
    log("Master switch disabled, skipping")
    return
  
  // 2. Fetch new listings from Airtable
  lastSync = getLastSyncTimestamp()
  listings = fetchAirtableListings(since=lastSync)
  updateLastSyncTimestamp()
  
  // 3. Filter already processed
  processedIds = getProcessedListingIds()
  unprocessed = listings.filter(l => !processedIds.includes(l.id))
  
  // 4. Get enabled switches sorted by priority
  enabledSwitches = getSwitches(is_enabled=true).orderBy('priority', 'asc')
  
  // 5. Evaluate each listing
  for listing in unprocessed:
    matchedSwitch = null
    
    for switch in enabledSwitches:
      if matchesCriteria(listing, switch.criteria):
        matchedSwitch = switch
        break  // Stop at first (highest priority) match
    
    if matchedSwitch:
      // Queue for generation
      createGenerationJob(listing, matchedSwitch)
      markProcessed(listing.id, matchedSwitch.id, status="queued")
      logGeneration(listing, matchedSwitch, "initiated")
    else:
      markProcessed(listing.id, null, status="skipped", reason="no_match")

function matchesCriteria(listing, criteria):
  // Price range check
  if criteria.minPrice and listing.price < criteria.minPrice:
    return false
  if criteria.maxPrice and listing.price > criteria.maxPrice:
    return false
  
  // State check
  if criteria.states and criteria.states.length > 0:
    if listing.state not in criteria.states:
      return false
  
  // Suburb check
  if criteria.suburbs and criteria.suburbs.length > 0:
    if listing.suburb not in criteria.suburbs:
      return false
  
  // Property type check
  if criteria.propertyTypes and criteria.propertyTypes.length > 0:
    if listing.propertyType not in criteria.propertyTypes:
      return false
  
  // Bedroom check
  if criteria.minBedrooms and listing.bedrooms < criteria.minBedrooms:
    return false
  
  // New build check
  if criteria.newBuildOnly and not listing.isNewBuild:
    return false
  
  return true

#### 7.2.1 Module 1: Authentication & Session Management

**Purpose**: Secure user authentication and session lifecycle management.

**Technical Implementation**:

**Login Flow**:
```
1. User submits username/password via login form
2. Frontend calls handle-login edge function
3. Edge function:
   a. Query custom_users by username
   b. Verify password hash using bcrypt
   c. Check user is_active status
   d. Generate cryptographically secure session token
   e. Create session record with expiration
   f. Set HTTP-only cookie with session_id
   g. Return session token and user data
4. Frontend stores session_token in sessionStorage
5. Frontend redirects to dashboard
```

**Session Token Generation**:
```typescript
const generateSessionToken = (): string => {
  const array = new Uint8Array(32);
  crypto.getRandomValues(array);
  return Array.from(array, byte => 
    byte.toString(16).padStart(2, '0')
  ).join('');
};
```

**Session Validation**:
Every authenticated request validates the session:
```typescript
const validateSession = async (token: string) => {
  const { data: session } = await supabase
    .from('user_sessions')
    .select('user_id, expires_at, last_activity_at')
    .eq('session_token', token)
    .maybeSingle();
  
  if (!session) return { valid: false, error: 'Invalid session' };
  if (new Date(session.expires_at) < new Date()) {
    return { valid: false, error: 'Session expired' };
  }
  
  // Update last activity
  await supabase
    .from('user_sessions')
    .update({ last_activity_at: new Date().toISOString() })
    .eq('session_token', token);
  
  return { valid: true, userId: session.user_id };
};
```

**Password Security**:
- Passwords hashed using bcrypt with cost factor 12
- Minimum password requirements enforced (8 chars, mixed case, numbers)
- Password reset tokens expire after 1 hour
- Failed login attempts logged for security monitoring

**Session Configuration**:
| Parameter | Value | Purpose |
|-----------|-------|---------|
| Token Length | 64 hex chars | Prevent brute force |
| Session Duration | 24 hours | Balance security/usability |
| Cookie Flags | HttpOnly, Secure, SameSite=Lax | XSS protection |
| Max Sessions | 5 per user | Limit concurrent access |
