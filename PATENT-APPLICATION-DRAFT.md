## 1. TITLE OF INVENTION

**INTEGRATED PROPERTY INVESTMENT ANALYSIS AND CLIENT MANAGEMENT SYSTEM WITH AI-DRIVEN REPORT GENERATION, MULTI-SOURCE DATA AGGREGATION, AND SECURE SERVICE-MEDIATED ARCHITECTURE**

## 2. ABSTRACT

A computer-implemented system and method for automated property investment analysis comprising:

A multi-tier report generation engine that produces comprehensive investment analysis documents through a twelve-section chunked processing architecture, wherein each section is independently generated by artificial intelligence and persisted progressively to enable fault-tolerant resumption without content regeneration;

A unified data aggregation pipeline that integrates real-time information from at least ten (10) disparate authoritative data sources including government census databases, crime statistics bureaus, meteorological services, property market indices, environmental risk registries, and financial rate providers, wherein said data is cached with configurable expiration and quality indicators;

A client relationship management subsystem that captures multi-dimensional investor profiles including detailed income breakdowns, categorized expenses, multi-property portfolios with individual cash flow tracking, and computed borrowing capacity assessments using lender-standard formulas with stress-testing;

A secure service-mediated database architecture wherein all frontend data operations are routed through authenticated serverless functions that verify user sessions and execute database queries using elevated privileges, preventing direct client-side database access while maintaining row-level security as a secondary enforcement layer;

A conversational document analysis interface utilizing Retrieval-Augmented Generation (RAG) for natural language querying of generated investment reports with streaming response delivery;

An automation engine with configurable trigger switches that automatically generate investment reports when new property listings matching specified criteria are detected.

The system addresses the technical problems of: (1) reducing property investment due diligence from hours of manual analysis to minutes of automated processing; (2) eliminating data fragmentation across multiple specialized platforms; (3) providing unified client intelligence combining property analysis with borrowing capacity assessment; and (4) enabling secure multi-user access to sensitive financial data through a novel mediation architecture.

## 3. TECHNICAL FIELD

The present invention relates generally to computer-implemented systems for real estate investment analysis and, more particularly, to an integrated platform that combines:

1. **Artificial Intelligence Document Generation**: Systems employing large language models for automated creation of structured investment analysis documents from property listing data and aggregated market information.

2. **Multi-Source Data Integration**: Methods for real-time aggregation of property-relevant data from disparate governmental, commercial, and environmental databases into unified analytical outputs.

3. **Financial Calculation Engines**: Computational systems for property investment metrics including yield analysis, cash flow projections, depreciation schedules, stamp duty calculations, and borrowing capacity assessments.

4. **Client Relationship Management**: Database systems for tracking investor client profiles, property portfolios, and investment readiness indicators.

5. **Secure Application Architecture**: Novel patterns for mediating database access in web applications through authenticated serverless function layers.

6. **Conversational AI Interfaces**: Systems for natural language interaction with generated documents using retrieval-augmented generation.

The invention is particularly applicable to the Australian property investment market, though the underlying architecture and methods are adaptable to other jurisdictions with appropriate data source substitution.

## 4. BACKGROUND OF THE INVENTION

### 4.1 Prior Art and Existing Solutions

The property investment analysis market currently comprises several categories of existing solutions, each addressing partial aspects of the investment due diligence workflow:

#### 4.1.1 Property Data Platforms

Existing platforms such as CoreLogic RP Data, PropertyData.com.au, and similar services provide access to property market data including:
- Historical sales records
- Median price trends
- Suburb demographic summaries
- Basic property attributes

**Limitation**: These platforms function as data repositories requiring manual interpretation. They do not generate synthesized investment analysis documents nor combine data with investor-specific financial circumstances.

#### 4.1.2 Investment Calculators

Various online calculators exist for individual investment metrics:
- Rental yield calculators
- Stamp duty calculators
- Loan repayment calculators
- Depreciation estimators

**Limitation**: These operate in isolation, requiring users to manually transfer outputs between tools. They do not account for investor-specific financial profiles or aggregate results into comprehensive analysis.

#### 4.1.3 Customer Relationship Management (CRM) Systems

General-purpose CRM platforms (Salesforce, HubSpot, industry-specific solutions like Rex) provide:
- Contact management
- Pipeline tracking
- Communication logging

**Limitation**: These systems are not designed for property investment workflows. They lack integration with property data sources, cannot perform investment calculations, and do not generate investment analysis documents.

#### 4.1.4 Report Generation Tools

Generic document generation tools and templates exist for creating property reports:
- Microsoft Word templates
- PDF form builders
- Basic automation tools

**Limitation**: These require extensive manual data entry, do not integrate with live data sources, and produce static documents without computational analysis.

### 4.2 Technical Problems Addressed

The present invention addresses the following technical problems not adequately solved by prior art:

#### Problem 1: Time-Intensive Manual Analysis

**Current State**: Preparing a comprehensive property investment due diligence report requires 4-8 hours of manual work including:
- Accessing 8-12 separate data platforms
- Transcribing data into spreadsheets
- Performing financial calculations
- Writing narrative analysis
- Formatting final documents

**Inventive Solution**: Automated report generation reducing the process to 5-15 minutes through AI-driven content synthesis and integrated data pipelines.

#### Problem 2: Data Fragmentation

**Current State**: Relevant investment data is dispersed across:
- Australian Bureau of Statistics (demographics)
- State crime statistics bureaus (safety data)
- Bureau of Meteorology (climate data)
- State RFS agencies (bushfire risk)
- Flood mapping services (flood risk)
- Domain/REA (market data)
- RBA (interest rates)
- State revenue offices (stamp duty rates)
- School directories (education data)
- Transport authorities (accessibility data)

**Inventive Solution**: Unified data aggregation layer with intelligent caching, quality indicators, and automated expiration management.

#### Problem 3: Disconnected Client Intelligence

**Current State**: Existing solutions treat property analysis and client financial profiling as separate activities. There is no mechanism to:
- Calculate borrowing capacity using detailed client income/expense data
- Assess portfolio-level performance across multiple properties
- Score investment readiness considering both property metrics and client circumstances

**Inventive Solution**: Integrated client relationship management with property portfolio tracking, borrowing capacity computation, and unified scoring systems.

#### Problem 4: Serverless Timeout Constraints

**Current State**: Complex AI document generation exceeds typical serverless function execution limits (30-150 seconds), causing incomplete outputs.

**Inventive Solution**: Chunked processing architecture with progressive persistence and fault-tolerant resumption.

#### Problem 5: Secure Multi-User Data Access

**Current State**: Web applications face security challenges with direct database access, requiring complex Row-Level Security policies that are difficult to maintain and audit.

**Inventive Solution**: Service-mediated architecture where all data operations flow through authenticated serverless functions using elevated privileges, with RLS as backup enforcement.

### 4.3 Objects of the Invention

It is an object of the present invention to provide a computer-implemented system that:

1. Generates comprehensive property investment analysis reports automatically using artificial intelligence, reducing preparation time by at least 90% compared to manual methods.

2. Aggregates real-time data from at least ten (10) authoritative data sources into unified investment analysis without requiring users to access multiple platforms.

3. Integrates property analysis with detailed client financial profiling to provide personalized investment assessments including borrowing capacity calculations.

4. Implements a fault-tolerant report generation architecture that can resume from interruption points without regenerating completed content.

5. Provides secure multi-user access to sensitive financial data through a mediated architecture that prevents direct database manipulation.

6. Enables natural language interaction with generated documents through conversational AI interfaces.

7. Automates report generation based on configurable triggers when new property listings matching specified criteria are detected.

## 5. SUMMARY OF THE INVENTION

### 5.1 Overview

The present invention provides an integrated property investment analysis and client management system implemented as a web-based platform comprising interconnected software modules executing on distributed computing infrastructure.

### 5.2 Core Architectural Components

The system comprises five (5) primary architectural layers:

#### Layer 1: User Interface Layer
A browser-based frontend application built using reactive component architecture (React framework) that provides:
- Dashboard views for system overview
- Report generation interfaces with manual override capabilities
- Client management interfaces with tabbed profile sections
- Pipeline visualization for sales/onboarding tracking
- Conversational AI chat interfaces
- Administrative control panels

#### Layer 2: API Gateway Layer
Serverless edge functions (Deno runtime) that:
- Authenticate user sessions via HTTP-only cookies or header tokens
- Validate user permissions for requested operations
- Mediate all database access using service-role credentials
- Integrate with external AI services (Perplexity, OpenAI)
- Integrate with external data providers
- Return sanitized responses to the frontend

#### Layer 3: Data Persistence Layer
A PostgreSQL database with:
- 78+ tables organized by functional domain
- Row-Level Security policies as secondary enforcement
- JSONB columns for flexible schema extension
- Automated trigger functions for computed fields
- Cache tables with expiration management

#### Layer 4: External Integration Layer
Connections to external services including:
- AI services for content generation
- Government data APIs
- Property market data providers
- CRM synchronization (GoHighLevel)
- Email services (Microsoft Graph)
- Voice AI services (Vapi)

#### Layer 5: File Storage Layer
Object storage for:
- Generated PDF reports
- Client documents
- Branding assets
- Template resources

### 5.3 Novel Features

The system implements the following novel features that distinguish it from prior art:

#### 5.3.1 Twelve-Section Chunked AI Generation

Investment reports are generated through a structured twelve-section architecture where:
- Each section is independently prompted and generated
- Section content is persisted to the database immediately upon generation
- A progress tracker (`last_completed_section`) records completion status
- If generation is interrupted, the system resumes from the last completed section
- Previous section content (4,000 characters) is injected into subsequent prompts to maintain narrative consistency

This approach overcomes serverless timeout limitations while ensuring document coherence.

#### 5.3.2 Multi-Source Risk Aggregation

Property risk assessment aggregates data from multiple independent sources:
- **Flood Risk**: State flood mapping services with risk classification
- **Bushfire Risk**: State RFS agencies with BAL (Bushfire Attack Level) ratings
- **Crime Statistics**: State bureaus of crime statistics with per-capita rates
- **Climate Data**: Bureau of Meteorology with historical patterns and projections
- **Environmental Hazards**: Council planning overlays and constraints

Each source is cached independently with source-specific expiration periods and quality indicators.

#### 5.3.3 Lender-Standard Borrowing Capacity Computation

The system calculates borrowing capacity using formulas derived from Australian lending practices:
- Income shading rules (reduction factors by income type)
- HEM (Household Expenditure Measure) benchmark comparison
- Assessment rate buffer application (+3%)
- Debt-to-Income (DTI) ratio calculation
- Stress-tested capacity computation
- Serviceability band classification

#### 5.3.4 Service-Mediated Security Architecture

All database operations are routed through a centralized invocation pattern:

```
Frontend → invokeSecureFunction() → Edge Function → Database
```

The edge function:
1. Extracts session token from HTTP-only cookie or `x-session-token` header
2. Validates session against `user_sessions` table
3. Verifies user permissions for the requested operation
4. Executes database query using `service_role` credentials
5. Returns sanitized response

This pattern prevents direct client-side database access while enabling flexible permission management.

#### 5.3.5 Automation Switch Architecture

Report generation can be automated through configurable "switches":
- Each switch defines matching criteria (price range, location, property type)
- A master switch enables/disables all automation
- When new listings are detected, each switch evaluates the listing
- Matching listings trigger automatic report generation
- Processing history prevents duplicate generation

#### 7.2.1 Module 1: Authentication & User Management

**Purpose**: Secure user authentication and granular permission management.

**Technical Implementation**:

The authentication system implements custom session-based authentication rather than relying on third-party authentication providers. This provides:
- Full control over session lifetime and management
- Custom role hierarchy (Superadmin → Admin)
- Granular per-module permissions (View, Edit, Delete)

**Session Lifecycle**:

1. **Login**: User submits credentials to `custom-auth-login` edge function
2. **Verification**: Bcrypt comparison of password hash
3. **Session Creation**: New record in `user_sessions` with unique token
4. **Cookie Setting**: HTTP-only cookie set with session ID
5. **Validation**: Subsequent requests validated via `custom-auth-verify`
6. **Logout**: Session deleted from database, cookie cleared

**Permission Model**:

| Role | Capabilities |
|------|-------------|
| Superadmin | All system access, user management, integrations |
| Admin | Module-specific access per permission grants |

Each admin user has a `user_permissions` record per module specifying:
- `can_view`: Access to view module content
- `can_edit`: Ability to create/modify records
- `can_delete`: Ability to remove records

#### 7.2.2 Module 2: Investment Report Generation

**Purpose**: Automated generation of comprehensive property investment analysis reports through AI-driven content synthesis.

**Novel Technical Features**:

**a) Twelve-Section Chunked Architecture**

Reports are divided into twelve (12) predefined sections:

| Section | Content |
|---------|---------|
| 1 | Executive Summary, Property Overview |
| 2 | Location Analysis, Suburb Profile |
| 3 | Property Features, Building Specifications |
| 4 | Market Analysis, Comparable Sales |
| 5 | Financial Analysis, Purchase Costs |
| 6 | Rental Assessment, Yield Analysis |
| 7 | Cash Flow Projections (10-Year) |
| 8 | Demographics, Population Trends |
| 9 | Infrastructure & Amenities |
| 10 | Risk Assessment (Flood, Bushfire, Crime) |
| 11 | Risks & Recommendations, Conclusion |
| 12 | Dynamic Template-Driven Sections |

Each section is:
1. Independently prompted with section-specific instructions
2. Generated by AI with token limits (2,000-5,000 tokens)
3. Immediately persisted to database upon generation
4. Tracked via `last_completed_section` field (0-11)

**b) Fault-Tolerant Resumption**

If generation is interrupted (network failure, timeout, user cancellation):
1. System records the last successfully completed section
2. Upon retry, system fetches existing content from database
3. Generation resumes from `last_completed_section + 1`
4. Previously generated content is preserved without regeneration

**c) Narrative Consistency Injection**

To maintain coherent narrative across sections:
1. After each section generation, content is stored
2. Before generating subsequent section, the last 4,000 characters of previous content are extracted
3. This context is injected into the prompt as "previousSections"
4. AI is instructed to maintain consistency with prior content

**Generation Algorithm** (Pseudocode):

```
function generateReport(propertyAddress, overrides, templateId):
  // 1. Create or fetch existing report record
  report = createReportRecord(propertyAddress, overrides)
  
  // 2. Determine starting section
  startSection = report.last_completed_section + 1
  
  // 3. Clear content if starting fresh
  if startSection == 1:
    report.report_content = ""
  
  // 4. Loop through sections
  for sectionNum from startSection to 12:
    // 4a. Build prompt with context
    previousContent = getLastNCharacters(report.report_content, 4000)
    prompt = buildSectionPrompt(sectionNum, propertyAddress, overrides, previousContent)
    
    // 4b. Call AI service
    sectionContent = callPerplexityAI(prompt, maxTokens=3000)
    
    // 4c. Append to report
    report.report_content += "\n\n" + sectionContent
    
    // 4d. Update progress tracker
    report.last_completed_section = sectionNum
    
    // 4e. Persist immediately
    saveReport(report)
  
  // 5. Mark complete
  report.status = "completed"
  saveReport(report)
  
  return report
```

**Report Tiers**:

The system supports three report tiers:

| Tier | Name | Length | Generation Method |
|------|------|--------|------------------|
| 1 | Investor's Compass | 50+ pages | Direct AI generation |
| 2 | Executive Briefing | ~20 pages | Condensation of Tier 1 |
| 3 | Snapshot | 4-5 pages | Condensation of Tier 2 |

Tier 2 and 3 reports are generated by passing the Tier 1 content to a condensation prompt with length constraints.

#### 7.2.5 Module 5: Borrowing Capacity Assessment

**Purpose**: Calculate maximum borrowing capacity using lender-standard formulas.

**Calculation Methodology**:

**Step 1: Income Aggregation and Shading**

Different income types are "shaded" (reduced) according to lender policies:

| Income Type | Shading Factor |
|-------------|---------------|
| Base Salary | 100% |
| Overtime (essential) | 80% |
| Overtime (non-essential) | 50% |
| Bonus | 50-80% (tenure-based) |
| Commission | 60% |
| Rental Income | 80% |

**Step 2: Expense Assessment**

The system compares declared expenses against HEM (Household Expenditure Measure) benchmarks:

```
if declared_expenses > HEM_benchmark:
  use declared_expenses
else:
  use HEM_benchmark
```

**Step 3: Surplus Calculation**

```
monthly_surplus = (shaded_annual_income / 12) - living_expenses - existing_commitments
```

**Step 4: Maximum Borrowing Calculation**

Using assessment rate (current rate + 3% buffer):

```
max_loan = PV(assessment_rate, loan_term_years * 12, monthly_surplus)
```

Where PV is the present value formula for an annuity.

**Step 5: Serviceability Band Classification**

| DTI Ratio | Band |
|-----------|------|
| < 4.0 | Excellent |
| 4.0 - 5.5 | Good |
| 5.5 - 7.0 | Marginal |
| > 7.0 | Restricted |

**Processing Logic**:

```
function processNewListings():
  // 1. Check master switch
  if not masterSwitch.is_enabled:
    return
  
  // 2. Fetch new listings from Airtable
  listings = fetchAirtableListings(since=lastSyncTime)
  
  // 3. Filter already processed
  unprocessed = filterProcessed(listings)
  
  // 4. Evaluate each listing
  for listing in unprocessed:
    matchedSwitch = null
    
    for switch in enabledSwitches.orderBy(priority):
      if matchesCriteria(listing, switch.criteria):
        matchedSwitch = switch
        break
    
    if matchedSwitch:
      // Queue for generation
      queueReportGeneration(listing, matchedSwitch)
      markProcessed(listing, matchedSwitch, status="queued")
    else:
      markProcessed(listing, null, status="skipped", reason="no_match")
```

### 7.3 Security Architecture Detail

#### 7.3.1 Service-Mediated Pattern

The core security innovation is the service-mediated database access pattern:

**Principle**: No frontend code directly accesses the database. All operations route through authenticated edge functions.

**Implementation**:

Frontend utility function:
```typescript
async function invokeSecureFunction(functionName: string, payload: object) {
  const sessionToken = sessionStorage.getItem('session_token');
  
  return await supabase.functions.invoke(functionName, {
    body: { ...payload, session_token: sessionToken },
    headers: { 'x-session-token': sessionToken }
  });
}
```

Edge function pattern:
```typescript
// 1. Extract session token
const sessionToken = 
  req.headers.get('x-session-token') ||
  cookies.get('session_id') ||
  body.session_token;

// 2. Validate session
const { data: session } = await supabase
  .from('user_sessions')
  .select('user_id, expires_at')
  .eq('session_token', sessionToken)
  .single();

if (!session || session.expires_at < new Date()) {
  return new Response(JSON.stringify({ error: 'Unauthorized' }), { status: 401 });
}

// 3. Get user and permissions
const { data: user } = await supabase
  .from('custom_users')
  .select('*, user_permissions(*)')
  .eq('id', session.user_id)
  .single();

// 4. Check permission for this operation
if (user.role !== 'superadmin') {
  const permission = user.user_permissions.find(p => p.module_key === moduleKey);
  if (!permission?.can_view) {
    return new Response(JSON.stringify({ error: 'Forbidden' }), { status: 403 });
  }
}

// 5. Execute with service_role (bypasses RLS)
const serviceClient = createClient(SUPABASE_URL, SERVICE_ROLE_KEY);
const { data, error } = await serviceClient.from(table).select();

// 6. Return sanitized response
return new Response(JSON.stringify({ data }), { status: 200 });
```

#### 7.3.2 Row-Level Security as Backup

RLS policies provide defense-in-depth:

```sql
-- All tables have RLS enabled
ALTER TABLE clients ENABLE ROW LEVEL SECURITY;

-- Service role bypasses RLS (used by edge functions)
-- This is built-in behavior

-- Backup policies for defense-in-depth
CREATE POLICY "Service role access only"
ON clients
FOR ALL
TO authenticated
USING (false);  -- Deny all direct authenticated access

-- Actual access is via service_role through edge functions
```

#### 7.3.3 Session Management

**Session Creation**:
- Unique token generated via cryptographically secure random generator
- Stored in database with user reference and expiration
- HTTP-only cookie set to prevent XSS access
- Session token also returned for cross-origin scenarios (stored in sessionStorage)

**Session Validation**:
- Every edge function request validates session
- Token checked against database
- Expiration verified
- User and permissions fetched

**Session Termination**:
- Explicit logout deletes session from database
- Cookie cleared
- SessionStorage cleared on client
