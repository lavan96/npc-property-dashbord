# Live Model Hub → Front-End Sync

Goal: whenever an AI model is changed for an agent via Model Hub, every front-end surface that shows or picks that model reflects the change immediately, without touching which back-end function actually fires. Also expand Report Q&A from a single hardcoded picker to a 3–4 slot layout, all governed by Model Hub.

Backend behaviour stays identical — edge functions keep resolving models through `agent_model_assignments` via `_shared/llmRouter.ts`. Only the *display* and *selection surfaces* change.

---

## Root cause of the current bug

`ModelSelector.tsx` and `ModelBadge.tsx` are hardcoded to four legacy providers (`openai` = "GPT-5.2", `openai-direct` = "GPT-4.1", `perplexity`, `gemini`). They never read `agent_model_assignments`, so when Model Hub swaps the `report_qa_*` agent to `openai/gpt-5.6`, the chip still says "GPT-5.2". The router at the edge is already correct — it's a UI-only drift problem, systemic across every surface that renders a model chip.

- **User picker** (dropdowns letting the user pick per-turn)

Known surfaces to cover (non-exhaustive, confirmed by the scan):

## Phase 1 — Shared model catalog + resolver hook (foundation)

New files:
- `src/lib/modelHub/agentKeys.ts` — canonical enum of every `agent_key` used across the app (single source of truth, mirrors DB seed).
- `src/lib/modelHub/modelDisplay.ts` — pure formatter: given a `model_id` (`openai/gpt-5.6`, `google/gemini-3-flash-preview`, `sonar-pro`, …) return `{ family, displayName, shortName, brand: BrandProfile, Icon }`. Reuses `src/lib/integrations/brandProfiles.ts` + `BrandMark` from the Integrations facelift. No hardcoded per-model strings — derives label from `model_id` + optional catalog cache row.
- `src/hooks/useAgentModels.ts` — React Query hook wrapping `agent_model_assignments`:
  - `useAgentModel(agentKey)` → single assignment (route, model_id, fallback_chain, last_used_at).
  - `useAgentModels(agentKeys[])` → batched fetch for pages that show several chips.
  - Realtime: subscribes to `postgres_changes` on `agent_model_assignments` and invalidates the query so *every* mounted chip re-renders within ~1s of a Model Hub save. Cleanup via `removeChannel` per the realtime standard.
  - Read path uses `invokeSecureFunction('agent-models-read', …)` (new thin edge fn) so we don't have to widen RLS on `agent_model_assignments`. Edge fn returns only the safe display fields.

New edge function `agent-models-read` (registered in `config.toml`, `verify_jwt = true`): accepts `{ agent_keys: string[] }`, returns the display-safe rows. No writes.

## Phase 2 — Universal display components

Replace the hardcoded `ModelBadge` / `ModelSwitchDivider` with model-hub-driven equivalents:

- `src/components/model-hub/LiveModelBadge.tsx` — props: `agentKey` (preferred) OR `modelId` (for per-message historical chips). Uses `useAgentModel` + `modelDisplay`. Renders brand icon + short name + tooltip ("Powered by {family} {model} · assigned via Model Hub"). Aurora-gold styling to match the existing chip language.
- `src/components/model-hub/LiveModelChipGroup.tsx` — used on multi-slot surfaces (Report QA) to show 3–4 chips at once.
- `src/components/model-hub/ModelUpgradeButton.tsx` — small "Upgrade" affordance that deep-links to Model Hub → Agent Bindings, prefiltered by `agentKey`. When the surface has multiple slots, one button offers "Upgrade all N slots".

Legacy `ModelBadge` becomes a thin wrapper that forwards to `LiveModelBadge` with a `modelId` prop, so old per-message chips keep working while new call sites move to `agentKey`.

## Phase 3 — Report Q&A: multi-slot picker (the concrete example)

Extend `agent_model_assignments` seed with three additional Report Q&A slots (schema unchanged — just more rows):
- `report_qa_primary` (current default agent)
- `report_qa_fast` (streaming / low-latency)
- `report_qa_deep` (long-context reasoning)
- `report_qa_search` (web-grounded, defaults to `sonar-pro`)

Frontend:
- Rewrite `ModelSelector.tsx` to pull the four slots via `useAgentModels`, render each with `LiveModelBadge`, and let the user pick which *slot* to use for the next question — not which raw provider. The slot → agent_key mapping is what flows to the backend in `sendMessage`.
- `ReportQA.tsx` state changes from `selectedModel: ModelProvider` to `selectedAgentKey: AgentKey`. The value passed to `report-qa` edge fn changes from `modelProvider` to `agentKey`; the edge fn already resolves the actual model via `llmRouter.load(agentKey)`, so no back-end logic changes — only the parameter name.
- Per-message `ModelBadge` now reads `agent_key` + `resolved_model_id` (persisted in `report_qa_messages.metadata`) so historical messages still render the exact model they used, while live chips reflect Model Hub in real time.
- Add a "Upgrade all Report Q&A slots" button on the header that jumps to Model Hub filtered to the four `report_qa_*` keys.

Back-compat: if `modelProvider` still arrives from an old client, the edge fn maps `openai → report_qa_primary`, `perplexity → report_qa_search`, etc., so no in-flight sessions break.

## Phase 4 — Roll the live badge across every other surface

Replace hardcoded chips one surface at a time using the inventory from Phase 0. Each replacement is a mechanical swap:
- Delete the hardcoded provider name / icon.
- Drop in `<LiveModelBadge agentKey="…" />`.
- Where the surface shows the model on a historical record (e.g. an old generated report), pass `modelId={record.model_id}` instead.

Priority order: Aurixa Agent chat header → Market Updates Ask AI → Agent Plans / Skills → Investment/Comparison/Portfolio/Suburb report footers → Email Copilot → PDF import / vision → Image generation. Each surface gets a matching `ModelUpgradeButton` where an admin is likely to want to swap the model.

## Phase 5 — Model Hub write path → instant fan-out

`ModelHub.tsx` already writes through `manage-agent-models`. Two small additions:
1. After a successful `update` or `bulk_update`, invalidate the `useAgentModels` query cache locally *and* rely on the realtime subscription to fan out to every other open tab / user session.
2. Toast confirmation names every surface that just refreshed ("Updated 4 Report Q&A slots — 6 UI chips refreshed").

Enable realtime on `agent_model_assignments` via migration (`ALTER PUBLICATION supabase_realtime ADD TABLE public.agent_model_assignments`). RLS is already `service_role`-only, so the realtime channel is opened by the new `agent-models-read` edge fn's session and filtered server-side.

## Phase 6 — Multi-slot generalization

Once Report Q&A proves the pattern, the same 3–4-slot layout is offered as an opt-in for any agent that benefits: Aurixa Agent (fast / deep / search / vision), Market Updates Ask AI (fast / deep / search), Image gen (default / hi-res). Turning it on for a new agent is just seeding extra `agent_model_assignments` rows and passing the new `agentKey` list to `<LiveModelChipGroup>`. Zero new components.
