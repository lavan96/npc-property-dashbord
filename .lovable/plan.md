## What "flattened" means here

A flattened PDF is one where every page has been **rasterised into an image** — no selectable text, no editable text/image overlays, no form fields, no annotation layers. This matches how the template-builder treats imported flattened PDFs (the trigger for this whole request). Use case: locked-down, "print-ready" handoff copies where the recipient can't extract or edit the underlying text.

## Approach — one shared utility, additive everywhere

I will **not** rewrite any existing PDF pipeline. The current "Download PDF" button on every surface keeps doing exactly what it does today. I'll add a second action next to it that re-uses the same generated bytes and runs them through a flatten step.

### Phase 1 — Shared infrastructure (3 new files)

1. `bun add pdfjs-dist` (required to rasterise pages; `pdf-lib`, `jspdf`, `html2canvas` already present).
2. `src/lib/pdf/flattenPdf.ts` — `flattenPdfBlob(blob, { dpi = 150 }): Promise<Blob>`. Loads the PDF with pdf.js, renders each page to an off-screen canvas, embeds the JPEGs into a fresh `pdf-lib` document at the original page size. Preserves page count, dimensions, orientation. Image-only output.
3. `src/lib/pdf/downloadPdf.ts` — `triggerPdfDownload(blob, filename)` standardised anchor-click helper (replaces the 20+ inline copies long-term; used by the new flatten button now).
4. `src/components/common/PdfDownloadMenu.tsx` — a small `<DropdownMenu>` wrapper. Accepts `{ getPdfBlob: () => Promise<Blob>, filename: string, label?: string }` and renders one trigger button with two items: "Download PDF" (default) and "Download as Flattened PDF". Used where the current button is a single download button. For surfaces that already have a dropdown (e.g. `ReportActionMenu`, `VownetPDFGenerator`, `Agreements`), I'll add a new `<DropdownMenuItem>` instead of replacing the trigger.

### UX

Where the existing trigger is a single button, it becomes a split button with a small chevron — left side runs the original action unchanged, dropdown adds "Download as Flattened PDF". Where the trigger is already a dropdown, I add one new menu item underneath the existing Download. Loading state on the flatten action shows "Flattening…".

### Risks & mitigations

- **Bundle size**: pdfjs-dist is ~1MB. The flatten utility will be lazy-imported (`await import('pdfjs-dist')`) so it only loads when a user clicks the flatten action.
- **Cold-start latency on large reports**: 150 DPI render of a 50-page report ≈ 10-25s in the browser. Progress toast will surface this.
- **Cross-origin worker**: pdfjs needs its worker. I'll use the bundled `pdf.worker.min.mjs` via Vite `?url` import to avoid CDN dependencies.
- **Server-rendered PDFs** (WeasyPrint, market-intelligence): the client receives a signed URL → fetches Blob → can flatten the same way. No backend changes needed.

## Technical notes

- pdfjs render: `getDocument(arrayBuffer).promise → page.getViewport({scale: dpi/72}) → page.render({canvasContext, viewport})`.
- Use JPEG quality 0.85; embed via `pdfDoc.embedJpg`. Preserves smaller file sizes vs PNG by ~5x on text-heavy pages.
- Set the new pdf-lib page size to `[viewport.width * 72/dpi, viewport.height * 72/dpi]` so physical page size stays identical.
- Worker setup: `import workerSrc from 'pdfjs-dist/build/pdf.worker.min.mjs?url'; GlobalWorkerOptions.workerSrc = workerSrc;`
